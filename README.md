# Домашнее задание к занятию «Микросервисы: подходы»

## Задача 1: Обеспечение разработки (CI/CD)

### Предлагаемое решение: **GitLab** (Self-managed) + **GitLab Runners**

#### Состав решения
- **GitLab** — единая платформа (Git-репозитории, CI/CD, реестр контейнеров, управление секретами).
- **GitLab Runners** — агенты сборки на собственных серверах с executor `docker` / `kubernetes`.
- **GitLab Container Registry** — хранение собственных Docker-образов.

#### Соответствие требованиям

| Требование | Реализация в GitLab |
|------------|----------------------|
| Облачная система | Развертывание в Yandex Cloud, AWS, GCP, Azure через Helm или официальные образы |
| Система контроля версий Git | Встроенный Git-репозиторий |
| Репозиторий на каждый сервис | Один проект / репозиторий на сервис |
| Запуск сборки по событию из VCS | Триггеры: `push`, `merge_request`, `tags` |
| Запуск сборки по кнопке с указанием параметров | **Pipeline schedules** или **Run pipeline** с переменными |
| Привязка настроек к каждой сборке | Каждый pipeline имеет свой контекст, переменные, окружение |
| Создание шаблонов для различных конфигураций сборок | `include` / `extends` / `!reference` в `.gitlab-ci.yml` |
| Безопасное хранение секретов (пароли, ключи доступа) | **CI/CD Variables** (зашифрованы, masked/protected), интеграция с HashiCorp Vault |
| Несколько конфигураций для сборки из одного репозитория | Разные jobs, rules, variables, parallel матрицы |
| Кастомные шаги при сборке | Произвольные `script`, `before_script`, `after_script` |
| Собственные Docker-образы для сборки проектов | `image: my-registry/custom-builder:latest` |
| Развернуть агентов сборки на собственных серверах | **GitLab Runner** с токеном, подключается к GitLab через REST/gRPC |
| Параллельный запуск нескольких сборок | Настройка `concurrent` у Runner, параллельные jobs в пайплайне |
| Параллельный запуск тестов | `parallel: matrix`, split по тестам, `needs` |

#### Обоснование выбора
- **Единый стек** — не требуется интеграция Jenkins + Nexus + Artifactory + Vault.
- **Native поддержка Docker и Kubernetes** — изолированные и воспроизводимые сборки.
- **Безопасность** — защита веток, маскирование переменных, аудит действий.
- **Масштабируемость** — горизонтальное масштабирование GitLab Runner под пиковые нагрузки.
- **Стандарт для микросервисной архитектуры** — отличная поддержка GitOps, Helm, CD-триггеров.

## Задача 2: Логи

### Предлагаемое решение: **Loki + Promtail + Grafana**

#### Состав решения
- **Promtail** — агент на каждом хосте, сбор логов из `stdout` контейнеров.
- **Loki** — центральное хранилище логов (индексация только по меткам).
- **Grafana** — пользовательский интерфейс, поиск, фильтрация.

#### Соответствие требованиям

| Требование | Реализация |
|------------|------------|
| Сбор логов в центральное хранилище со всех хостов | Promtail → Loki (по HTTP/GRPC) |
| Минимальные требования к приложениям, сбор логов из stdout | Приложение пишет в stdout, Docker/containerd перенаправляет, Promtail подхватывает |
| Гарантированная доставка логов до центрального хранилища | Promtail использует дисковую очередь (`boltdb`) при потере связи с Loki |
| Обеспечение поиска и фильтрации по записям логов | **LogQL** — фильтры по меткам + тексту (регэксп, contains, JSON) |
| Пользовательский интерфейс с доступом для разработчиков | **Grafana Explore** — удобный интерфейс с автодополнением LogQL |
| Возможность дать ссылку на сохранённый поиск по записям логов | Сохранение запроса в дашборд, копирование URL (абсолютная/относительная ссылка) |

#### Обоснование выбора
- **Экономия ресурсов** — в отличие от ELK, Loki не индексирует тело сообщения, только метки. Экономия дисков и памяти в 10–20 раз.
- **Единый интерфейс с мониторингом** — Grafana как единая точка входа для логов и метрик (см. Задачу 3).
- **Работа с Kubernetes** — нативный discovery подов через Promtail.
- **Гарантия доставки** — Promtail сохраняет позиции чтения и буферизует данные при отказе Loki.
- **Простота для разработчиков** — не нужно писать сложные парсеры.

## Задача 3: Мониторинг

### Предлагаемое решение: **Prometheus + Node Exporter + cAdvisor + Grafana**

#### Состав решения
- **Prometheus** — сбор и хранение метрик (pull-модель).
- **Node Exporter** — метрики хоста (CPU, RAM, HDD, Network).
- **cAdvisor** — метрики контейнеров/сервисов (потребление ресурсов).
- **Grafana** — визуализация, дашборды, агрегация.

#### Соответствие требованиям

| Требование | Реализация |
|------------|------------|
| Сбор метрик со всех хостов, обслуживающих систему | Prometheus опрашивает Node Exporter на каждом хосте |
| Сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network | Node Exporter (`node_cpu_seconds_total`, `node_memory_MemAvailable`, `node_filesystem_*`) |
| Сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network | cAdvisor (`container_cpu_usage_seconds_total`, `container_memory_working_set_bytes`) |
| Сбор метрик, специфичных для каждого сервиса | Prometheus client library в коде сервиса (`http_requests_total`, `db_query_duration_seconds`) |
| Пользовательский интерфейс с возможностью запросов и агрегации | Prometheus Web UI или Grafana Explore (PromQL) |
| Пользовательский интерфейс с настраиваемыми панелями для отслеживания состояния системы | **Grafana** — готовые дашборды (Node Exporter, Kubernetes), создание кастомных панелей |

#### Обоснование выбора
- **Индустриальный стандарт** — Prometheus + Grafana де-факто стандарт для микросервисов (CNCF, используется в K8s).
- **Pull-модель** — упрощает обнаружение проблем с доставкой метрик, встроенный service discovery.
- **Высокая производительность** — собственная TSDB, возможность горизонтального масштабирования через Thanos/Cortex.
- **Гибкие алерты** — Alertmanager для нотификаций (Telegram, Slack, PagerDuty).
- **Единый стек с логами** — общая Grafana для метрик и логов (Prometheus + Loki).

## Итоговая архитектура

| Компонент           | Инструменты                                            |
| ------------------- | ------------------------------------------------------ |
| CI/CD + Git         | GitLab (self-managed в облаке) + GitLab Runners       |
| Логи                | Loki + Promtail + Grafana                             |
| Метрики мониторинга | Prometheus + Node Exporter + cAdvisor + Grafana       |

### Развертывание
- GitLab, Prometheus, Loki, Grafana — в облаке (Yandex Cloud / AWS / GCP) на выделенных ВМ или в Managed-версиях.
- Node Exporter, Promtail, cAdvisor — на всех хостах с сервисами (через Docker Compose, Ansible или DaemonSet в Kubernetes).
- GitLab Runners — на собственных серверах компании.

### Преимущества предложенного подхода
1. **Открытость** — все компоненты с открытым исходным кодом, нет vendor lock-in.
2. **Единая экосистема** — Grafana как единое окно для логов и метрик, GitLab для всего CI/CD.
3. **Масштабируемость** — горизонтальное масштабирование каждого компонента.
4. **Безопасность** — управление секретами, аудит, защита веток.
5. **Готовность к Kubernetes** — все инструменты имеют нативную поддержку K8s.
